SQL Injection

1. Web Application Vulnerability – SQL Injection

A user enters a malicious input inside a login form. Because the website does not validate or sanitize the input, the database interprets the text as a command instead of normal data. As a result, confidential information may be revealed.

How It Happens

  • Occurs when user input is directly inserted into SQL queries.
  • No input validation or sanitization.
  • Database responses reveal too much information.

    • Impact

  • Use prepared statements (parameterized queries).
  • Apply strict input validation.
  • Use stored procedures.
  • Enable Web Application Firewall (WAF).

    • Insecure Login API

    2. Mobile Application Vulnerability – Insecure Login API

    A mobile app sends login credentials through an unprotected connection. An ethical tester intercepts the traffic using Burp Suite or MITMproxy and sees the username and password in plain text.

    How It Happens

  • API does not use HTTPS.
  • Sensitive data is sent without encryption.
  • Weak or no authentication tokens.

    • Impact

  • Account takeover.
  • Unauthorized access to user data.
  • Replay attacks.

    • Prevention

  • Enforce HTTPS/TLS encryption.
  • Use token-based authentication (JWT/OAuth).
  • Validate certificates.

    • Insecure Data Storage

    3. Android Device Vulnerability – Insecure Data Storage

    An Android app stores sensitive information such as tokens, passwords, or chats in plain text. Using ADB, an ethical tester accesses the device storage and views the unencrypted data easily.

    How It Happens

  • Storing sensitive data in plain text.
  • Using insecure storage locations.
  • No encryption or hashing.

    • Impact

  • Exposure of personal data.
  • Identity theft.
  • Session hijacking.

    • Prevention

  • Encrypt local storage.
  • Use Android Keystore.
  • Avoid storing sensitive data unless necessary.

    • File Permission Misconfiguration

    4. Desktop Application Vulnerability – File Permission Misconfiguration

    A Windows desktop application stores configuration or credential files in a publicly accessible folder. Anyone connected to the network can access or modify these files.

    How It Happens

  • Incorrect file permissions.
  • Storing sensitive data in shared network folders.
  • Weak access controls.

    • Impact

  • Unauthorized file access.
  • Manipulation of configuration files.
  • Privilege escalation.

    • Prevention

  • Use proper NTFS file permissions.
  • Avoid exposing internal files.
  • Encrypt sensitive configuration files.

    • IoT Device Vulnerability

    5. IoT Device Vulnerability – Default Credentials

    An IoT router or camera or router uses a default username and password. An ethical tester scans the device and finds unsecured services like Telnet or SSH, then logs in using known default credentials.

    How It Happens

  • Device is shipped with common default passwords.
  • No forced password change.
  • Exposed remote access services.

    • Impact

  • Full device takeover.
  • Spying (for cameras).
  • Adding device to botnets (e.g., Mirai).

    • Prevention

  • Force password changes on first setup.
  • Disable unnecessary services.
  • Apply firmware updates.
  • Use strong password policies.

    • Phishing Attack

    6. Social Engineering Vulnerability – Phishing Attack

    An employee receives an email that looks identical to the company's official IT support message. The email says: “Your account will be locked. Click the link below to verify your identity.” The link opens a fake login page that perfectly copies the original website. When the employee enters their username and password, the attacker immediately receives the credentials.

    How It Happens

  • Attackers create fake websites that mimic real login portals.
  • Victims are tricked into clicking malicious links in:
      ◦ Emails
      ◦ SMS
      ◦ WhatsApp
      ◦ Fake ads
  • Attackers use social engineering to create urgency or fear.
  • No technical vulnerability is required — it targets humans, not systems.

    • Impact

  • Stolen usernames and passwords.
  • Full account takeover (email, banking, social media).
  • Data breaches inside organizations.
  • Financial loss or unauthorized transactions.
  • Malware delivery (keyloggers, ransomware).
  • Identity theft.

    • Prevention

  • User awareness training — recognize suspicious messages.
  • Always check:
      ◦ Sender email
      ◦ URL domain
      ◦ HTTPS certificate
  • Enable Multi-Factor Authentication (MFA).
  • Use email filtering and anti-phishing protection.
  • Never click unknown links — navigate manually.
  • Use browser protection tools against malicious websites.
